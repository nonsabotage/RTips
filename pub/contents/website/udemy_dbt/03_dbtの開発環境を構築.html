03 DBTの開発環境を構築しよう

July 25, 2025

1 dbtの概要

1.1 dbtの利点

  1. Jinjaテンプレートエンジンを使用してSQLを拡張
  2. YAMLファイルでモデルのメタデータを管理
  3. ソフトウェア開発の手法をデータ分析に適用

1.2 公式サンプルデータセット

  • jaffle shop

1.3 3層構成

  • seeds
    • raw
  • staging
    • stg
  • marts
    • fact/dim

2 使い方

2.1 dbt

2.1.1 ビルド

jaffle_shopにカレントディレクトリを移動して、dbtを実行します

poetry run dbt run --profiles-dir .

特定のテーブルだけを更新するとき

poetry run dbt run --select my_model

2.1.2 テスト

テストを実行したい。modelsの下に、schema.yamlを作成して、テストを定義します。

# すべてのテストを実行
poetry run dbt test --profiles-dir .

# 特定のモデルのテストのみ実行
poetry run dbt test --select stg_customers --profiles-dir .

カスタムのテストをおこないたいときは、testsディレクトリにSQLファイルを作成します。

poetry run dbt test --select assert_order_has_items --profiles-dir .

dbtのカスタムテストでは「例外」を返す形で定義する。 つまり、期待値は0であるようなクエリを定義して、そのクエリがレコードを返すようだとテストが失敗ということになります。

2.1.3 ドキュメント

ドキュメントを生成するときには次のコマンドを実行する。

poetry run dbt docs generate --profiles-dir .

サーバーの立ち上げ

poetry run dbt docs serve --profiles-dir .

2.1.4 ソース

dbtで追加したわけでないテーブルについては、ソースとして、schema.yamlに定義します。 そのときに, databaseやschemaを指定する必要があり, それに対して, 名前が付くという形になります。

カラム定義はテスト用なので必須ではない。

# ソースが正しく定義されているか確認
poetry run dbt list --resource-type source --profiles-dir .

# ソースデータの鮮度を確認（freshness設定がある場合）
poetry run dbt source freshness --profiles-dir .

# ソース定義に含まれるテストを実行
poetry run dbt test --select "source:*" --profiles-dir .

2.1.5 インクリメンタル

# 初回実行（フルリフレッシュ）
poetry run dbt run --select fact_orders_incremental --profiles-dir .

# 2回目以降の実行（増分のみ）
poetry run dbt run --select fact_orders_incremental --profiles-dir .

# 強制的にフルリフレッシュ
poetry run dbt run --select fact_orders_incremental --full-refresh --profiles-dir .

2.1.6 マクロとコンパイル

マクロを定義して、SQLの中で使うことができます。


{{
    config(
        materialized='table'
    )
}}

select
    store_id,
    {% for product_type in get_product_types() %}
    count(distinct case when p.type = '{{ product_type }}' then i.sku end) as {{ product_type }}_product_count
    {% if not loop.last %},{% endif %}
    {% endfor %}
from {{ ref('stg_orders') }} o
join {{ ref('stg_items') }} i on o.order_id = i.order_id
join {{ ref('stg_products') }} p on i.product_sku = p.product_sku
group by store_id

コンパイルをおこなう

poetry run dbt compile --select product_count_by_store --profiles-dir .

2.1.7 dbt-Utils

dbt-utilsを使うと、dbt側で作成された便利なマクロが使えます。

packages.ymlを作成する. 次を実行するとUtilsがインストールされます。

dbt deps

2.1.8 スナップショット

スナップショットを作成することで, マスタの変更履歴などを管理することができる

{% snapshot customers_snapshot %}

{{
    config(
      target_schema='snapshots',
      unique_key='id',
      strategy='check',
      check_cols=['name'],
    )
}}

select
    *
from {{ source('jaffle_shop_raw', 'customers') }}

{% endsnapshot %}
poetry run dbt snapshot --profiles-dir .

2.2 poetry

poetryのバージョンがおかしいとき

poetryの依存関係がおかしいとき, pyprojectを変数して, 一旦ロックして, installし直す

poetry lock
poetry install

3 dbt_project.yml

dbtはプロジェクトの設定と、プロファイルの設定を分けて管理します。

dbtプロジェクトの設定は、dbt_project.ymlに記述します。

name: 'jaffle_shop'
version: '1.0.0'
config-version: 2
profile: 'jaffle_shop'
source-paths: ['models']
analysis-paths: ['analysis']
test-paths: ['tests']
data-paths: ['data']
macro-paths: ['macros']
snapshot-paths: ['snapshots']
target-path: 'target'
clean-targets:

    - 'target'
    - 'dbt_modules'
models:
    jaffle_shop:
        +materialized: view
        staging:
            +materialized: view
        marts:
            +materialized: table
        seeds:
            +materialized: table
        snapshots:
            +materialized: incremental
        tests:
            +materialized: table
        analysis:
            +materialized: table
        macros:
            +materialized: macro
        sources:
            +materialized: source
        custom:
            +materialized: table
